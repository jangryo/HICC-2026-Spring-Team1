## 웹서버(WEB) vs 웹어플리케이션(WAS)
웹은 HTTP 기반이다
- HTML, 이미지, 파일, JSON 등 거의 모든 형태의 데이터 전송 가능
- 클라이언트 <-> 서버 뿐만 아니라 서버 <-> 서버 간의 데이터 전달도 HTTP를 이용함

**WEB**은 정적 리소스를 제공한다. 
**WAS**는 웹 서버 기능에 어플리케이션 로직을 수행할 수 있다. 그러나 웹서버도 프로그램을 실행하는 기능을 포함하기에, 경계가 모호하다.

## 웹 시스템 구성법
웹 시스템을 구성할때 WEB과 WAS의 균형을 잡을 필요가 있다.  
만약 **WAS**와 **DB**로만 웹 시스템을 구성한다면 **WAS**가 많은 역할을 담당해야하며, 서버 과부화가 나타날 우려가 있다. 또한 장애 발생 시 어플리케이션이 멈춰버리기 떄문에, 오류 화면을 출력할 수 없다.

따라서 **WEB**, **WAS**, **DB**를 함께 구성해야한다. 정적 리소스는 **WEB**이 처리하고, 동적 처리가 필요한 경우 **WAS**가 애플리케이션 로직 처리를 담당한다. **WAS** 장애가 발생하더라도, **WEB**을 통해 오류 화면을 출력할 수 있게 된다.

정적리소스가 많이 사용되는 경우 **WEB**을, 어플리케이션 리소스가 많이 사용된다면 **WAS**를 증설하면 된다.

## 서블릿
**서블릿**은 HTTP 요청 및 응답 정보를 편리하게 사용 할 수 있게 해준다.  
HTTP 요청 메시지
```
POST /save HTTP/1.1
Host: localhost:8080
Content-Type: application/x-www-form-urlencoded
username=kim&age=20
```
HTTP 응답 메시지
```
HTTP/1.1 200 OK
Content-Type: text/html;charset=UTF-8
Content-Length: 3423
<html>
 <body>...</body>
</html>
```

- 서버 TCP/IP 연결 대기, 소켓 연결
- HTTP 요청 메시지 파생해서 읽기
- POST 방식, /save URL 인지 ......

웹 어플리케이션을 직접 구현하려면 많은 것을 해야하지만 **서블릿**을 이용하면 '의미있는 비즈니스 로직'만 수행하면 된다.

```java
@WebServlet(name = "helloServlet", urlPatterns = "/hello")
public class HelloServlet extends HttpServlet {
 @Override
 protected void service(HttpServletRequest request, HttpServletResponse response){
 //애플리케이션 로직
```

localhost:8000/hello URL에 접속하면 코드가 실행된다. request 파라미터를 통해 요청 정보를, response 파라미터를 통해 응답 정보를 쉽게 입력 할 수 있다.

## 서블릿 컨테이너
서블릿을 지원하는 WAS를 **서블릿 컨테이너**라고 한다. 서블릿 컨테이너는 서블릿 객체의 생명주기를 관리한다. 서블릿 객체는 싱글톤으로 관리한다. 필요할때마다 객체를 생성하는 것보다 재활용하는 것이 더 효율적이기 때문이다. 그러나 싱글톤으로 관리하기 떄문에 **공유 변수** 사용에 주의해야한다.  

## 멀티 쓰레드
**쓰레드**는 어플리케이션의 코드를 순차적으로 실행하는 것을 말하는데, 동시 처리가 필요하다면 쓰레드를 추가로 생성해야한다. 단일 요청에서 쓰레드를 하나 사용할때, WAS에서 쓰레드를 할당하여 서블릿을 호출하여 응답하게 된다. 하지만 다중 요청에서는 쓰레드가 이미 사용중이기 때문에, 기존의 요청이 끝날때까지 대기해야한다. 이를 해결하기 위해 요청 마다 쓰레드를 생성할 수도 있는데, 하나의 쓰레드가 지연되어도 나머지 쓰레드도 정상적으로 동작한다. 하지만 쓰레드 생성 비용이 매우 비싸고, 요청이 너무 많이 오면 임계점을 넘어 서버가 다운될 수도 있다.  

이 단점들을 해결하기 위해 **쓰레드 풀**을 사용할 수 있다. 필요한 쓰레드를 쓰레드 풀에 보관하고 관리한다. 쓰레드가 미리 생성되어 있으므로, 쓰레드를 생성하고 종료하는 비용이 절약되고, 응답시간이 빠른 장점이 있다. 최대 쓰레드 수(max thread) 설정이 중요하다 쓰레드를 낮게 설정하면, 서버의 CPU는 여유가 있지만, 클라이언트에서 쓰레드 대기 때문에 응답 지연이 자주 발생한다. 만약 너무 높으면 임계점 초과로 서버가 다운될 수도 있다.적정 숫자를 찾으려면, 아파치 ab, 제이미터 등의 성능 테스트를 시도하여 찾는다.

멀티 쓰레드에 대한 부분은 WAS가 처리하기 떄문에, 개발자는 관련 코드를 신경쓰지 않아도 된다.


## HTML, HTTP API, CSR, SSR
- 정적 리소스: HTML, CSS 등 이미 생성된 리소스 파일을 제공한다.
- HTML 페이지: WAS에서 동적으로 HTML을 생성하여 전달한다
- HTTP API: HTML이 아니라 JSON 형으로 데이터를 전달한다  
HTTP API는 웹 <-> 서버 뿐만아니라, 서버 <-> 서버에서도 이용된다. 전자인 경우 WAS가 DB에서 정보를 조회하여 데이터를 웹에 제공하고, 웹에서 UI를 업데이트한다. 후자의 경우 주문 서버 WAS에서 결제 서버 WAS를 호출하는 것이다.  
- CSR(클라이언트 사이드 렌더링): 서버에서 HTML, JS, 데이터를 제공한다  
웹 브라우저에서 HTML, JS, HTTP API 데이터를 제공하면 웹에서 JS로 HTML 결과를 보여준다.
- SSR(서버 사이드 렌더링): 서버에서 최종 HTML을 생성하여 전달한다
서버에서 DB에 데이터를 조회하고 이를 통해 동적으로 HTML을 생성하여 웹 브라우저에 전달한다.



CSR은 주로 웹 프론트엔드에서 이용된다. 백앤드의 경우 SSR을 알아야한다.  
백엔드 개발자는 서버, DB, 인프라 등 다양한 백엔드 기술을 공부해야한다. 해당 사항을 먼저 공부한 후 웹 프론트 기술 학습을 하는 것이 좋다.


## 자바 웹 기술 역사
현재 기술은 애노테이션 기반 스프링 MVC의 등장으로 개발이 훨씬 간결해졌으며, `@Controller`를 이용해 코드의 생산성이 크게 항상되었다. 스피링 부트의 등장으로 과거에는 서버에 WAS를 직접 설치하고, 소스는 War 파일을 만들어서 WAS에 배포해야했지만, 빌드 결과에 WAS를 포함하여 빌드 배포를 단순화하였다.

이후 스프링 웹 기술은 서블릿 기반의 Spring MVC, Reactive 기반의 Spring WebFlux로 분화되었다.WebFlux는 적은 쓰레드로 높은 성능을 낼 수 있고 효율적인 동시처리가 가능하다는 장점이 있지만, 기술적 난이도가 높고 RDB 지원이 부족하다. 그리고 MVC도 충분히 빠르기 때문에 실무에서는 사용 비중이 낮은 편이다
